# Ревью проекта и сверка с `malakhov-ai-digest-architecture.md`

Дата: 2026-04-16

## Короткий вывод

Проект уже сильно вырос относительно архитектурного документа. По факту это не только Next.js-сайт с RSS-агрегатором на Node, а гибрид из:

- legacy-слоя на `Next.js + Supabase articles + Node pipeline`
- нового основного слоя на `FastAPI + PostgreSQL + event-layer + scheduler + quality/reporting + HTML site shell`

То есть работа проделана большая, но единая картина проекта сейчас распалась. Главная проблема не в том, что “мало сделано”, а в том, что код, документация, схема БД и способы запуска описывают разные версии продукта.

## Что уже сделано по факту

### 1. Появился полноценный Python backend

Это уже не MVP из одного `articles`-стола.

- Есть FastAPI-приложение с health-check, internal/public API, HTML-страницами сайта и internal job endpoints: `app/api/main.py`.
- Есть отдельные сервисные слои: ingestion, normalization, clustering, classification, scoring, digest, alpha, quality, site rendering.
- Есть scheduler на APScheduler: `app/jobs/`.
- Есть Alembic-миграции и полноценные SQLAlchemy-модели: `app/db/models/`, `alembic/`.

Подтверждение:

- `AGENTS.md` уже фиксирует целевую архитектуру как `Python 3.12 + FastAPI + PostgreSQL + Alembic + aiogram + APScheduler` (`AGENTS.md:14-31`).
- `pyproject.toml` требует Python `>=3.12` и описывает FastAPI-стек (`pyproject.toml:5-38`).

### 2. Реализован event-layer вместо старой схемы `articles`

Архитектурный документ говорит: “не делаем events как отдельную сущность”, но по факту именно они и стали основой продукта.

- Есть `raw_items`, `sources`, `source_runs`, `events`, `event_sources`, `event_categories`, `event_tags`, `digest_issues`, `digest_issue_items`, `alpha_entries`, `process_runs`, `llm_usage_logs`.
- В `ProcessEventsService` уже есть нормализация, кластеризация, первичный источник, скоринг, shortlist для LLM, summary и связь с предыдущими событиями.

Это сильный шаг вперёд по качеству продукта: стало меньше “агрегаторности” и больше редакторской логики.

### 3. Есть сайт нового поколения поверх FastAPI

Новая версия сайта уже не завязана только на Next.js.

- FastAPI сам отдаёт HTML-страницы `/`, `/events`, `/events/{slug}`, `/issues`, `/issues/{id}`, `/russia`, `/alpha`, `/sitemap.xml` (`app/api/main.py:1724-1909`).
- HTML-сайт уже учитывает редакционные ограничения: фильтрация слабых и псевдо-материалов, связанный контент, issue-навигация, отдельный “ИИ в России”, “Альфа”.
- В `app/web/site.py` видно, что продуктовая идея сайта уже стала намного богаче исходного документа (`app/web/site.py:21-80` и дальше).

### 4. Есть тестовый слой, и он заметный

В репозитории хороший тестовый периметр:

- API-тесты
- тесты миграций
- тесты shortlist/scoring/source policy
- тесты digest builder/delivery
- тесты public site shell

То есть проект уже не выглядит как прототип без валидации. Это плюс.

## Что устарело или расходится с документом

### 1. `malakhov-ai-digest-architecture.md` больше не является “единой точкой истины”

Документ обещает:

- один стек `Next.js + Node.js + Supabase + GitHub Actions`
- одну таблицу `articles`
- отсутствие `events`
- отсутствие APScheduler и отдельного backend

Но текущий код это уже опровергает:

- в документе сайт живёт на Next.js/Vercel и читает `articles` (`malakhov-ai-digest-architecture.md:22-134`)
- в коде основной runtime уже Python/FastAPI (`AGENTS.md:14-31`, `pyproject.toml:5-38`)
- в коде публичный сайт отдаёт сам API (`app/api/main.py:1724-1909`)

Фактически документ описывает старый MVP, а не текущий проект.

### 2. `README.md` тоже описывает старую версию проекта

Сейчас README говорит:

- ставим только `npm install`
- применяем `supabase/schema.sql`
- запускаем `npm run dev`
- вся логика построена вокруг Next.js и Node pipeline (`README.md:15-76`)

Но реальный проект уже требует:

- Python 3.12
- зависимости Python
- Alembic
- PostgreSQL для нового слоя
- запуск FastAPI/бота/scheduler

То есть новый разработчик по README почти гарантированно поднимет не тот стек.

### 3. `supabase/schema.sql` устарел и описывает только legacy-мир

Файл создаёт только таблицу `articles` (`supabase/schema.sql:4-51`).

Проблема:

- README просит применить именно этот SQL
- но Python backend работает не на этой схеме, а на множестве таблиц через Alembic

Итог: человек может честно выполнить README, получить “успешно созданную БД”, но backend всё равно не заработает.

### 4. В репозитории одновременно живут две несовместимые архитектуры

Есть legacy Node-слой:

- `pipeline/*.ts`
- `bot/daily-digest.ts`
- Next pages, читающие `articles`
- Supabase schema для `articles`

Есть новый Python-слой:

- `app/api`, `app/services`, `app/db`, `app/jobs`
- event-layer
- digest issues
- HTML site shell

Сами по себе оба слоя полезны как история перехода, но вместе в одном активном репозитории без явного статуса они создают путаницу.

## Что не работает или находится в зоне риска

### 1. Локальный запуск сейчас не воспроизводим по документации

Проверка дала такие результаты:

- `pytest -q` не запускается, потому что `pytest` не установлен в текущем shell
- системный Python на машине: `3.9.6`
- проект требует `Python >=3.12` (`pyproject.toml:10`)
- `uv` в системе отсутствует

Следствие простым языком:

Сейчас репозиторий нельзя считать “самоподнимаемым” по текущей документации. Чтобы просто проверить проект, человеку придётся самому угадывать правильную среду.

### 2. Frontend build в текущей среде падает сразу

`npm run build` упал с ошибкой:

- установлен `Node v18.15.0`
- Next.js требует `>=18.17.0`

При этом в `package.json` нет `engines`, которые заранее подсветили бы это ограничение (`package.json:1-44`).

Следствие:

Локальная сборка ломается ещё до проверки логики приложения.

### 3. В репозитории есть явная фронтенд-заглушка `TEST OK`

Файл `src/app/page.tsx` содержит:

```tsx
export default function Home() {
  return <div>TEST OK</div>
}
```

Это видно в `src/app/page.tsx:1-3`.

Проблема не только в самой заглушке, а в том, что рядом существует другой `app/page.tsx`, где лежит уже “реальная” страница (`app/page.tsx:1-37`). Это признак незавершённого или плохо зафиксированного перехода между структурами `app/` и `src/app/`.

### 4. Workflow-слой тоже раздвоен

Старые workflow:

- `rss-parse.yml` гоняет Node ingest в `articles` (`.github/workflows/rss-parse.yml:1-34`)
- `enrich.yml`
- `tg-digest.yml`

Новые workflow:

- `daily_digest.yml` будит Render API и гоняет Python ingestion/process/build/send (`.github/workflows/daily_digest.yml:1-57`)
- `weekly_digest.yml`

Если это всё оставить активным одновременно, высок риск:

- двойных cron-задач
- параллельной жизни двух пайплайнов
- расхождения между тем, что лежит в `articles`, и тем, что реально публикуется через event-layer

### 5. `.env.example` не соответствует правилу “только плейсхолдеры”

В `.env.example` лежит реальный-looking `TELEGRAM_CHANNEL_ID=-1003966448216` (`.env.example:1-9`).

Это не секрет в строгом смысле, но для шаблона окружения лучше хранить только нейтральные значения. Сейчас это противоречит собственному правилу из `AGENTS.md`, что `.env.example` должен содержать placeholders (`AGENTS.md:55-59`).

### 6. Жёстко зашиты deployment-specific значения

Примеры:

- `BASE_URL: https://malakhov-ai-digest-api.onrender.com` в workflow (`.github/workflows/daily_digest.yml:11-13`)
- `_SITE_BASE_URL = "https://news.malakhovai.ru"` в `app/web/site.py:21`
- жёстко зашитый Yandex Metrika ID в шаблоне сайта

Это делает переносимость и тестирование хуже. Если меняется домен/инстанс/окружение, менять надо код и workflow, а не только env.

## Главные технические выводы

### Сильные стороны

- Хорошо продвинут новый редакционный слой.
- Есть event-layer вместо плоского агрегатора.
- Есть quality/reporting и внятный internal API.
- Есть широкое покрытие тестами.
- Продуктовая идея стала заметно сильнее исходного MVP.

### Слабые стороны

- Нет одного канонического способа запускать проект.
- README, архитектура, схема БД и код рассказывают разные истории.
- Legacy Node/Next слой не отделён формально от нового Python/FastAPI слоя.
- Операционный слой завязан на hardcoded deployment-значения.

## Что я бы доработал в первую очередь

### Приоритет 1. Определить каноническую архитектуру

Нужно принять одно решение и зафиксировать его в документах:

1. Python/FastAPI/event-layer — это основной продукт.
2. Node/Next/articles — либо legacy, либо экспериментальный слой.
3. Один source of truth для данных: либо `events`, либо `articles`, но не “оба как будто равноправны”.

Если текущий прод уже сидит на Python-слое, значит legacy-часть надо прямо так и пометить.

### Приоритет 2. Переписать README и архитектурный документ

Нужно разделить:

- `docs/architecture_current.md` — как проект устроен сейчас
- `docs/legacy_articles_stack.md` — что осталось от старого MVP, если нужно сохранить

В новый README надо вынести:

- требуемые версии `Python 3.12+` и `Node 20+`
- как поднять backend
- как применить Alembic
- как сидировать sources
- как запускать scheduler
- как запускать и тестировать сайт
- статус legacy Node-скриптов

### Приоритет 3. Убрать или заморозить legacy-дубли

Минимум нужно сделать одно из двух:

- либо удалить `src/app` как мусорный/старый слой
- либо архивировать legacy Next-часть в отдельную директорию вроде `legacy/next-site`

Сейчас наличие одновременно `app/page.tsx` и `src/app/page.tsx` — плохая практика и источник будущих ошибок.

### Приоритет 4. Привести БД и bootstrap в один сценарий

Сейчас есть два мира:

- `supabase/schema.sql`
- Alembic migrations

Нужно выбрать один путь:

- для актуального backend: только Alembic
- для legacy articles-пайплайна: отдельный legacy bootstrap

Иначе новый разработчик почти гарантированно поднимет неправильную схему.

### Приоритет 5. Убрать hardcoded deployment values в env/config

Вынести в env:

- public site base URL
- API base URL для workflows
- Yandex Metrika ID
- любые внешние каналы доставки

Тогда проект станет переносимым между `local / stage / prod`.

## Простыми словами: что тут происходит

Если совсем просто, то проект уже вырос из первоначальной идеи.

Раньше план был такой:

- собирать RSS
- перевести статьи
- сложить всё в `articles`
- показать карточки на сайте

Сейчас проект стал умнее:

- он собирает сырые материалы из источников
- нормализует их
- объединяет в события
- ранжирует
- собирает выпуски
- отдельно делает слой “ИИ в России”
- отдельно делает “Альфу”
- и уже сам умеет отдавать сайт через backend

Это хорошо. Но документы всё ещё рассказывают старую историю. Из-за этого человек видит один проект на бумаге, а в коде находит другой.

## Рекомендуемое решение

Лучший путь сейчас:

1. Признать Python/FastAPI/event-layer основным продуктом.
2. Перевести старый Node/Next/articles-слой в явный legacy-статус.
3. Переписать документацию под реальный код.
4. Упростить локальный запуск до одного понятного сценария.

Тогда проект станет не просто “мощным”, а ещё и управляемым.
