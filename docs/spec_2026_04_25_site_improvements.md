# Site Improvements — Spec for Claude Code

> Дата: 2026-04-25
> Источник: задачи владельца проекта на доработку news.malakhovai.ru
> Тип: временная спецификация. После реализации каждой волны — переносить итоги в канонические docs (`ARTICLE_SYSTEM.md`, `DESIGN.md`, `OPERATIONS.md`, `PROJECT.md`) и удалять из этого файла выполненные пункты.

---

## Контекст

Текущее состояние:
- Сайт работает на Next.js 15 + Supabase, тёмная тема по умолчанию.
- Главная и страницы топиков рендерятся, но визуально не соответствуют целевому редакционному стилю.
- Пагинации в разделах нет: показывается только одна страница, остальные статьи не видны пользователю.
- Cookie/согласие настроены под скрипт Метрики, а не по 152-ФЗ.
- Enrich-промпт не всегда генерирует таблицы и форматированные списки, хотя инфраструктура (`article_tables`, `editorial_body`) для этого есть.
- Один доменный лендинг (`malakhovai.ru`) и поддомен с новостями (`news.malakhovai.ru`) — дубликатов нет, но canonical/robots/sitemap/Метрика на news ещё не разделены явно.

Целевое состояние после трёх волн:
1. Фронт переработан под редакционный стиль: светлая тема по умолчанию, лента в духе VC.ru с одной горячей статьёй, чистые разделы без лишних боковых блоков.
2. Модель данных перестроена: одна основная категория + до двух смежных, чистые URL с категорией в пути, рабочая пагинация, корректные хлебные крошки.
3. Контент стал содержательнее: enrich стабильно делает таблицы и нумерованные списки, в `topics/ai-research` приходят более глубокие материалы, добавлен AI/стартап-фид VC.ru.

## Принципы работы для Claude Code

1. **Не объединять волны в один PR.** Каждая волна — отдельная ветка, отдельный PR. Внутри волны допустимы 2-3 коммита по подзадачам.
2. **Перед началом каждой задачи прогнать `npm run context`** и прочитать соответствующий канонический doc.
3. **Любое изменение поведения = обновление канонического doc в том же PR.** Без этого PR не считается завершённым (см. правило в `CLAUDE.md`).
4. **Не трогать `legacy/`.** Это заморожено.
5. **Не менять enrich-промпт в волнах 1 и 2.** Это работа волны 3.
6. **Не трогать модель `articles ↔ categories` в волне 1.** Фронт волны 1 строится на текущей структуре топиков. Модель меняется только в волне 2.
7. **После каждой волны — smoke-check на проде** по чек-листу из `OPERATIONS.md` плюс задачи-специфичный чек-лист (см. ниже в каждой волне).
8. **Если задача меняет публичные URL — в том же PR добавить редиректы** со старых URL на новые. Без редиректов SEO провалится.

# Волна 1 — Фронт и редакционный вид

**Цель:** сайт по умолчанию светлый, главная и страницы топиков переработаны под VC-подобный формат, лишние боковые блоки убраны, разделы кликабельны.

**Канонические docs для обновления:** `DESIGN.md`, `PROJECT.md` (если меняется product surface).

**Не трогать:** модель данных, URL-структуру `/articles/[slug]`, enrich-промпт, pipeline.

## 1.1. Светлая тема по умолчанию

**Что сделать:**
- В компоненте, который инициализирует тему (вероятно `app/layout.tsx` или ThemeProvider), сменить дефолт с `dark` на `light`.
- Убедиться, что `localStorage` не перезаписывает дефолт для новых посетителей.
- Существующих пользователей с сохранённым `theme: dark` не трогать — пусть остаются на тёмной.
- Проверить `prefers-color-scheme` поведение: если у пользователя в системе тёмная — что показываем? **Решение по умолчанию:** показываем светлую (наш дефолт), пользователь сам переключит при желании. Это редакционный сайт, мы хотим консистентного вида.

**Acceptance:**
- Открытие сайта в инкогнито на любом устройстве показывает светлую тему.
- Переключатель в хедере работает в обе стороны.
- Выбор пользователя сохраняется и переживает перезагрузку.

**Doc impact:** `DESIGN.md` — поправить раздел «Темы», явно указать «Светлая тема — default».

## 1.2. Лента главной страницы в духе VC.ru

**Референс:** скриншот VC.ru с кратким списком заголовков сверху + персональная статья ниже. Идея — не копировать визуал VC, а взять структуру: компактный список свежих заголовков как «пульс дня», крупная горячая статья отдельно, дальше остальная лента.

**Что сделать на десктопе:**
- В верхней секции главной — двухколоночный блок:
  - **Левая колонка (≈40% ширины):** список из 6-8 свежих статей одними заголовками + время добавления (например, «32 минуты назад», «2 часа назад»). Без обложек, без тизеров. Только заголовок, время, тег категории мелким шрифтом.
  - **Правая колонка (≈60% ширины):** одна горячая статья дня — большая обложка, заголовок, лид, кнопка «Читать».
- Под этой секцией — стандартная лента карточек как сейчас, но с обновлёнными стилями по `DESIGN.md`.

**Что сделать на мобильном:**
- Список свежих заголовков сверху (компактный, можно с иконкой времени).
- Под ним — горячая статья дня крупной карточкой.
- Дальше — обычная лента в одну колонку.

**Логика выбора горячей статьи:**
- Берём статью с наивысшим `score` за последние 24 часа из `published=true && verified_live=true`.
- Если ничьего score не дотягивает до порога (например, score < 5) — берём самую свежую опубликованную.
- Эта логика должна жить в `lib/articles.ts` как функция `getHotStoryOfTheDay()`.

**Логика «время назад»:**
- На клиенте: компонент `<RelativeTime date={...} />` использующий `Intl.RelativeTimeFormat` с русской локалью.
- На сервере для SSR: при первом рендере показываем абсолютное время («14:32»), на клиенте гидрация заменяет на относительное. Это избегает hydration mismatch.

**Acceptance:**
- На десктопе видны обе колонки в шапке, на мобильном — стек.
- Список заголовков обновляется автоматически (через ISR / `revalidate`).
- Время отображается корректно и обновляется без перезагрузки страницы (раз в минуту достаточно).
- Выбор горячей статьи воспроизводим: если запустить getHotStoryOfTheDay дважды подряд — результат тот же.

**Doc impact:** `DESIGN.md` — добавить раздел «Главная: блок дня». `PROJECT.md` — обновить описание главной поверхности.

## 1.3. Чистка страниц разделов (топиков)

**Что убрать:**
- Боковой блок с описанием темы (если есть). Описание оставить только в самом верху страницы как лид-параграф.
- Любые декоративные блоки, которые не несут навигационной или контентной нагрузки.

**Что сделать кликабельным и видимым:**
- В шапке каждой страницы топика — горизонтальный ряд тегов всех разделов, с подсветкой текущего. Каждый тег — `<Link>` на свой раздел.
- На главной — такой же ряд под hero-блоком, чтобы пользователь сразу видел все разделы и мог переключаться.

**Дизайн разделов:**
- Привести в соответствие с `DESIGN.md` (теги, типографика, отступы).
- Заголовок раздела — крупный, по `DESIGN.md` H1 страницы категории (48px).
- Лид (1-2 предложения, что в этом разделе) — серым, под заголовком.

**Acceptance:**
- На странице любого топика виден полный список разделов в виде кликабельных тегов.
- Боковых пустых блоков нет.
- Переход между разделами не требует возврата на главную.

**Doc impact:** `DESIGN.md` — обновить раздел «Страница категории».

## 1.4. Куки-баннер и согласие по 152-ФЗ

**Контекст:** сейчас на сайте, по описанию, всплывает запрос на использование Яндекс.Метрики. Это неправильно — по российскому законодательству нужно спрашивать согласие на обработку персональных данных и cookies, а не на конкретный скрипт.

**Что нужно сделать:**

**Юридическая часть (тексты — заложить заглушки, потом дать юристу проверить):**
1. Создать страницу `/privacy-policy` — Политика обработки персональных данных. Шаблон по 152-ФЗ должен содержать: цели обработки, перечень обрабатываемых данных, правовые основания, сроки хранения, права субъекта, контакты ответственного за обработку.
2. Создать страницу `/cookie-policy` — Политика использования cookies. Перечислить категории cookies (необходимые / аналитические / маркетинговые), указать конкретные сервисы (Яндекс.Метрика).
3. Создать страницу `/consent` или раздел в политике — описание процедуры отзыва согласия.

**UI-часть:**
- Баннер при первом заходе: «Мы используем cookies для аналитики и улучшения сайта. Подробнее в Политике cookies. [Принять все] [Только необходимые] [Настроить]».
- При нажатии «Только необходимые» — Яндекс.Метрика и любая другая аналитика **не загружается**.
- При нажатии «Настроить» — модалка с переключателями по категориям.
- Решение пользователя сохраняется в `localStorage` под ключом, например, `consent_v1`. Версия в ключе нужна, чтобы при изменении политики можно было перезапросить согласие.

**Техническая часть:**
- Скрипт Яндекс.Метрики загружается **только** после получения согласия на аналитические cookies.
- Сейчас, скорее всего, Метрика грузится из `app/layout.tsx` безусловно — её нужно вынести в компонент `<Analytics />`, который проверяет `consent_v1` и грузит скрипт через `next/script` со стратегией `lazyOnload` уже после согласия.
- Логирование факта согласия (дата, версия политики, выбор пользователя) — в `localStorage` достаточно для MVP. Серверное логирование — отдельная задача на потом, если будет нужно.

**Важное предупреждение:**
Юридические тексты, которые сгенерирует Claude Code, — это **шаблон**, не финальный документ. Иван, перед публикацией покажи их юристу или используй проверенный шаблон с rkn.gov.ru / typical templates от Тинькофф/Яндекса для образца. В коде поставить TODO-комментарии в местах, где требуется юридическая правка.

**Acceptance:**
- При заходе в инкогнито виден баннер.
- При выборе «Только необходимые» Яндекс.Метрика **не появляется** в Network-вкладке DevTools.
- При выборе «Принять все» Метрика загружается.
- Страницы `/privacy-policy`, `/cookie-policy`, `/consent` доступны и проиндексируются.
- Ссылки на эти страницы есть в футере.

**Doc impact:** `PROJECT.md` — добавить страницы в product surfaces. `OPERATIONS.md` — упомянуть, что аналитика теперь условно-загружаемая.

## 2.1. Перестройка модели «статья ↔ категория» — DONE

> Реализовано в ветке `wave2-categories-model`. Канонические доки обновлены:
> `docs/ARTICLE_SYSTEM.md` (раздел Categories), `docs/ARCHITECTURE.md` (RLS contract +
> модель категорий), `docs/DECISIONS.md` (ADR-006). Миграция —
> `supabase/migrations/013_categories_model.sql`. Helper —
> `lib/categories.ts::splitTopicsToCategories`.
>
> Slug-и категорий на старте совпадают с прежними `topics` (`ai-industry`, `ai-research`,
> `ai-labs`, `ai-investments`, `ai-startups`, `ai-russia`, `coding`). Перенумерация —
> задача 2.2 вместе с редиректами.

**Текущая модель:** статья имеет массив `topics` (теги). Все равноправны.

**Новая модель:**
- В таблице `articles` добавить поле `primary_category` (text, NOT NULL, FK на справочник категорий).
- Добавить поле `secondary_categories` (text[], максимум 2 элемента, может быть пустым).
- Создать таблицу-справочник `categories` со столбцами: `slug`, `name_ru`, `description_ru`, `order_index`, `is_active`.
- Категории на старте — те же, что текущие топики плюс «Стартапы» (см. волну 3.4): `industry`, `research`, `labs`, `russia`, `code`, `startups`. Точный список Иван должен утвердить — заложить миграцию с возможностью добавлять.

**Миграция данных:**
- Для каждой существующей статьи — `primary_category` ← первый элемент текущего `topics`, `secondary_categories` ← остальные (максимум 2).
- Если у статьи нет topics — `primary_category` ← `industry` как дефолт. Залогировать такие случаи отдельно для ручной проверки.

**Backward compatibility:**
- Старое поле `topics` не удалять сразу. Оно остаётся read-only в течение волны 2 для возможного отката. Удаляется в отдельной миграции после полного зелёного цикла.

**Acceptance:**
- Все опубликованные статьи имеют валидный `primary_category`.
- В Supabase Dashboard видны новые колонки и таблица `categories`.
- Старые запросы по `topics` продолжают работать (на чтение) до явного cutover.

**Doc impact:** `ARTICLE_SYSTEM.md` (раздел Editorial fields), `ARCHITECTURE.md` (data contracts), `DECISIONS.md` (новый ADR — переход на primary + secondary).

- Старые URL из sitemap убрать (они и так теперь редиректят).

**Acceptance:**

**Doc impact:** `ARTICLE_SYSTEM.md` (раздел Slug policy → URL policy), `OPERATIONS.md` (smoke-check).

## 2.3. Пагинация в разделах

**Диагностика — сделать в первую очередь:**
1. Открыть `lib/articles.ts` и `app/topics/[slug]/page.tsx` (или его аналог после волны 2.2).
2. Найти запрос, который тянет статьи для раздела. Проверить наличие `LIMIT`, `OFFSET`, или `range()`.
3. **Самая вероятная причина:** есть `LIMIT 20` или похожее, нет offset. То есть запрос всегда возвращает те же 20 свежих, а пагинация не реализована.

**Реализация:**
- Параметр `?page=N` в URL.
- На сервере (Server Component) — `range((page-1)*20, page*20 - 1)` в Supabase запросе.
- Внизу страницы:
  - Кнопка «Показать ещё» при page < total_pages — на клиенте подгружает следующую страницу через RSC (`router.push`) или Server Action и **дописывает** к текущему списку (не перезагружает страницу).
  - Альтернатива: классические «Назад / Вперёд / 1 2 3 ...» под лентой.

**Решение по UX:**
По твоему запросу — «как удобнее и быстрее по скорости загрузки». Кнопка «Показать ещё» с догрузкой через Server Action — лучший компромисс: быстро, не перезагружает страницу, корректно работает SEO для первой страницы (она же по умолчанию). Делать её.

**Производительность:**
- Запросы на счётчик статей кешировать через `unstable_cache` или Supabase RPC с TTL 5 минут.
- На странице категории не загружать сразу 100 статей — 20 на первой странице, потом докручивать.

**Acceptance:**
- На странице раздела с >20 статьями виден счётчик («1-20 из 47») и кнопка «Показать ещё».
- При клике подгружаются следующие 20.
- При повторном клике — следующие 20.
- Когда статьи кончились — кнопка скрывается.
- URL меняется на `?page=2` при подгрузке (для шаринга).

**Doc impact:** `ARTICLE_SYSTEM.md` (раздел про рендер списков), `OPERATIONS.md` (smoke-check).

## 2.4. SEO-разделение malakhovai.ru и news.malakhovai.ru

**Контекст:** на главном домене лендинг про Ивана/проекты, на news — поддомен новостей. Дубликатов контента нет, но настроек, явно разделяющих SEO, скорее всего тоже нет.

**Что сделать на news.malakhovai.ru:**
1. **`robots.txt`** — убедиться, что есть и разрешает индексацию основных разделов, запрещает `/internal`, `/api`, `/_next`. Если нет — создать.
2. **`sitemap.xml`** — после волны 2.2 он уже на новой схеме URL. Убедиться, что он отдаётся по корню (`https://news.malakhovai.ru/sitemap.xml`) и зарегистрирован в Яндекс.Вебмастере и Google Search Console для **этого** домена, а не для главного.
3. **Canonical** — на каждой странице news canonical начинается с `https://news.malakhovai.ru/...`. Никаких ссылок на `https://malakhovai.ru` как canonical быть не должно.
4. **Open Graph и Twitter** — `og:url`, `twitter:url` на news всегда содержат поддомен.
5. **Проверка перелинковки с главного домена:**
   - На лендинге `malakhovai.ru` ссылки на новости должны вести явно на `https://news.malakhovai.ru/...` — без относительных путей.
   - С news на лендинг ссылки могут быть, но они не должны быть в каждом футере как «основной сайт» — лучше отдельным пунктом «Об авторе».
6. **Метрика и Search Console:**
   - На news должен быть **отдельный счётчик** Яндекс.Метрики, отдельный в Google Analytics (если используется).

   - В Google Search Console — то же самое.
7. **`hreflang` и `lang`** — на news явно `<html lang="ru">`. Если на лендинге есть английская версия — ничего не менять, просто проверить, что они никак не пересекаются по hreflang.

**Acceptance:**
- `https://news.malakhovai.ru/robots.txt` отдаётся и валиден.

# Волна 3 — Контент и enrich

**Цель:** статьи в `/categories/research` (бывший `topics/ai-research`) стали глубже; enrich стабильно генерирует таблицы и нумерованные списки; добавлен AI-фид VC.ru; раздел «Стартапы» получает альфа-материалы.

**Канонические docs:** `ARTICLE_SYSTEM.md`, `editorial_style_guide.md`, `feeds.config.ts`.

## 3.1. Улучшение enrich-промпта: таблицы и нумерация

**Контекст:** референсная статья — про Sequoia. У неё таблицы и блоки сделаны. У большинства других — нет, хотя у них тоже есть структурируемые данные. Значит, проблема в промпте, не в инфраструктуре.

**Что сделать в `pipeline/claude.ts` или `pipeline/enricher.ts`:**

1. **В промпт добавить явный раздел про таблицы:**
   ```
   Если в исходном материале есть структурированные данные, оформи их как таблицу в article_tables:
   - сравнения (X vs Y vs Z)
   - перечисления с одинаковыми атрибутами (модели + параметры + цена; компании + раунд + сумма)
   - временные оси (год + событие)
   - метрики (бенчмарк + score)

   Если структурированных данных нет — таблицу не делай. Не натягивай.
   ```

2. **Для нумерованных списков и «N направлений / N способов / N компаний»:**
   ```
   Если статья по сути перечисление (заголовок «10 направлений», «5 ключевых трендов»),
   оформи editorial_body так, чтобы цифры и пункты были явно выделены — заголовки
   подразделов с номерами или нумерованный список markdown.
   ```

3. **Тест-промпт на 20 статьях:**
   - Прогнать новый промпт на 20 уже опубликованных статьях разных типов (анонс модели, аналитика, перечисление, обычная новость).
   - Проверить вручную: где Claude добавил таблицу — она уместна? где не добавил — действительно нет данных?
   - Если ложноположительных >2/20 — ужесточить промпт.

4. **Промпт-контракт:**
   - Зафиксировать в `editorial_style_guide.md` правила: когда таблица уместна, когда нумерация, как именно их форматировать в `editorial_body` и `article_tables`.

**Acceptance:**
- На 20 тестовых статьях `article_tables` появляется только там, где это уместно.
- На статьях с цифрами в заголовке («10 направлений») — внутри тела чётко выделены пункты с номерами.
- Изменения промпта применены и в проде новые статьи начинают приходить с улучшенной структурой.

**Doc impact:** `ARTICLE_SYSTEM.md` (раздел Editorial fields), `editorial_style_guide.md` (новые правила).

## 3.2. Глубина материалов в /categories/research

**Контекст:** ты хочешь, чтобы статьи в research были более информативны и по тексту, и по картинкам. Можно меньше количество, но качественнее.

**Что менять:**

1. **Score threshold для research отдельный.**
   - Сейчас единый `score >= 2` для всех. Завести в `pipeline/scorer.ts` концепцию категорийного порога.
   - Для `research` — `score >= 4`. Для остальных — `>= 2`. Параметры держать в `pipeline/scorer.config.ts`.
2. **Fetcher строже на `research`.**
   - Если у статьи в research отсутствует cover image и в исходнике нет inline-картинок — отбрасывать с пометкой `rejected_low_visual`.
   - Это жёсткая мера. Альтернатива: помечать как `needs_image` и догенерировать через DALL-E. Но генерация — отдельный pipeline. Пока — отбрасывать.
3. **Минимальная длина текста для research.**
   - В quality gate enrich: для статей с `primary_category=research` требовать `editorial_body.length >= 1500` символов. Иначе reject.
4. **Расширить промпт enrich для research.**
   - Если статья — research, в промпт явно дописать: «это материал про исследования. Расскажи подробнее: какая проблема, какой подход, какие результаты, какие ограничения, что это меняет». Это даёт глубину автоматически.

**Acceptance:**
- В research новые статьи имеют либо cover, либо хотя бы одну inline-картинку.
- Текст ≥ 1500 символов в editorial_body.
- Поток в research уменьшится количественно (это ожидаемо), но качество вырастет.

**Doc impact:** `ARTICLE_SYSTEM.md` (раздел Score и publish gate), `editorial_style_guide.md`.

## 3.3. Добавление VC.ru как источника

**Контекст:** ты хочешь VC.ru как ресурс, особенно для альфы в стартапы. Прямой RSS всего сайта — это шум. Нужен тематический фид.

**Что сделать:**

1. **Исследовательский шаг (1 час):**
   - Найти у VC.ru тематические RSS-фиды. У них точно есть `/feeds/[topic]`.
   - Кандидаты: AI-фид, стартапы-фид, технологии-фид. Найти точные URL.
   - Прогнать каждый кандидат вручную через `pipeline/rss-parser.ts` за неделю — посмотреть, какой % материалов реально по теме AI/стартапов и какой шум.
2. **Если нашёлся хороший фид (>70% релевантности):**
   - Добавить в `pipeline/feeds.config.ts` с пометкой `source: 'vc.ru'`, `topics: ['startups', 'industry']`, `editorial_priority: 'high'` (опционально).
   - Если фид смешанный — добавить keyword-фильтр на этапе ingest: пропускать только статьи, в которых заголовок или description содержит AI/ИИ/стартап-связанные ключи.
3. **Если хорошего фида нет:**
   - Брать общий фид VC.ru плюс жёсткий keyword-фильтр (`AI`, `ИИ`, `нейросеть`, `LLM`, `OpenAI`, `Anthropic`, и т.д.).
   - Список ключей вынести в `pipeline/keyword-filters.ts`.

**Acceptance:**
- В Supabase появляются записи с `source='vc.ru'`.
- Спустя 7 дней оценить вручную: сколько из них прошли в публикацию, сколько — мусор. Если мусор >30% — ужесточить фильтр.

**Doc impact:** `ARTICLE_SYSTEM.md` (раздел источников), `OPERATIONS.md` (если меняется cron-настройка).

## 3.4. Раздел «Стартапы» с альфа-материалами

**Контекст:** ты хочешь альфа-инфу — крутые находки с рынка. Это категориальная задача: новый раздел + правила отбора.

## Глобальные риски и предупреждения

1. **Юридические тексты для куки-баннера — это шаблон, не финал.** Перед публикацией показать юристу.
2. **Редиректы со старых URL критичны.** Если в волне 2.2 их забыть — Search Console через неделю покажет резкий рост 404, и SEO провалится.
3. **Изменение модели категорий — необратимо без отката миграции.** Перед накаткой на прод сделать бэкап Supabase.
4. **Изменение enrich-промпта аффектит cost.** Если промпт стал длиннее — стоимость на статью вырастет. Проверить через `npm run cost:report` после первой недели работы новой версии.
5. **VC.ru может банить парсер** при слишком частых запросах. Соблюдать `User-Agent` и не парсить чаще раза в 30 минут.
