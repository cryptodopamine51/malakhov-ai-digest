## 1. Где сейчас уходят деньги (картина на 2026-06-10)

При полном выпуске 60–70 статей/день структура расхода выглядит так:

| Статья расхода | Порядок цифр | Комментарий |
|---|---|---|
| DeepSeek (основной writer) | **~$0.01/день** | Копейки. Трогать не нужно. |
| **Claude premium fallback** (Sonnet 4.6 Batch) | **~$3–5/день** | Главный потребитель. Habr AI один дал $2.48 за 81 вызов; лонгриды — $0.06+/статья. |
| OpenAI картинки (gpt-image-1.5 low) | ≤$1/день (hard cap) | $0.013/обложка, бюджет-гард уже стоит. |
| Claude reviewer (caption и пр.) | мелочь | |

**Вывод:** ~80% денег — это статьи, которые «выпали» из дешёвого DeepSeek-пути в полный
Claude-рерайт. Все главные рычаги экономии — вокруг того, **кто и почему туда попадает**
и **какой моделью** этот fallback обслуживается.

## 2. Экономия на тексте (главный блок)

### 2.1. Починить ложные срабатывания risk-флагов — бесплатная экономия

В cheap-режиме статья уходит в дорогой premium fallback, если у неё есть флаг
`research / legal_regulation / medical / geopolitics / high_score`. Флаги ставятся
регэкспами в `pipeline/editorial-routing.ts`, и они **ловят подстроки**, а не слова:

- `GEOPOLITICS_RE` содержит `выбор` → срабатывает на «выбор модели», «на выбор»,
  «выборка» — обычная бытовая лексика любой обзорной статьи;
- `LEGAL_RE` содержит `закон` → ловит «закономерность», «закономерно»;
- `LEGAL_RE` содержит `персональн` → ловит «персональный ассистент», «персональные
  рекомендации» — то есть половину статей про ИИ-ассистентов;
- `MEDICAL_RE` `клиник` / `врач` тоже стоит проверить на частотность.

Каждое ложное срабатывание = статья за ~$0.02–0.06 вместо ~$0.001. Фикс — границы слов
(Unicode `\P{L}`, как уже сделано в scorer) + точечные стемы («выборы», «закона/законопроект»,
«персональных данных» вместо голого «персональн»). Качество не страдает вообще: эти статьи
никогда не были юридическими/геополитическими.

**Перед фиксом замерить:** прогнать `detectEditorialRiskFlags` по последним 500 статьям и
посчитать долю каждого флага + примеры срабатываний. Подозрение: ложных там 30–60%.

### 2.2. `high_score` — рецензировать, а не переписывать

Сейчас флаг `high_score` (score ≥ 8) отправляет **лучшие** статьи в полный Claude-рерайт.
Логика понятна («топ-материалы — премиум-качество»), но экономика обратная: генерация —
это ~2 800 output-токенов по $15/MTok (≈$0.04+), а **рецензия** того же материала — это
~300 output-токенов (≈$0.002–0.005).

Предложение: high_score статьи вести по пути «DeepSeek пишет → компактный Claude reviewer
проверяет» (этот путь уже существует — режим `balanced` / `shouldReviewWithClaude`).
Premium-рерайт оставить только когда reviewer вернул `premium_fallback`. По сути — включить
для high_score «balanced», а не «premium». Качество защищено reviewer-ом, расход падает
в 5–10 раз на этих статьях.

### 2.3. Haiku 4.5 вместо Sonnet 4.6 там, где Sonnet избыточен

Цены (актуальные): Sonnet 4.6 — $3/$15 за MTok; **Haiku 4.5 — $1/$5** (в 3 раза дешевле,
с Batch-скидкой — $0.5/$2.5). Кандидаты на перевод:

- **компактный reviewer** (он и так «compact QA», задача — pass/fail JSON) — Haiku справится;
- **fallback-writer для не-research категорий** (новости с money/legal флагом — это всё ещё
  новостной пересказ, не глубокая аналитика);
- `ai-research` оставить на Sonnet — там реально нужна глубина (порог 1500 символов body).

Риск: стиль Haiku слабее. Митигировать так же, как с DeepSeek: строгий validator + выборочный
Sonnet-reviewer для Haiku-результатов на первые 1–2 недели, сравнить reject-rate.

### 2.4. Обрезать вход для premium-пути

`buildEditorialUserMessage` в `pipeline/claude.ts` шлёт `originalText` **целиком**, без капа.
DeepSeek-бриф режет исходник до 5 000 символов — и качество не жалуется. Habr-лонгриды на
30–60К символов раздувают input premium-вызова в разы (это и есть «$0.06+/статья» из аудита).
Предложение: кап исходника для premium-пути на ~12–15К символов (с обрезкой по границе
абзаца). Редакционный body всё равно 1 200–3 000 символов — хвост лонгрида в него не попадает.

### 2.5. Навести порядок в учёте: возможно, мы «тратим» вдвое меньше, чем видим

Известная с 2026-05-07 проблема: `estimateTextCostUsd` нигде в проде не вызывается с
`batch: true`, то есть Batch-скидка 50% **не применяется в логах** — cost-report и
budget-гард видят list price. Реальный биллинг Anthropic может быть ~вдвое ниже отчётного.
Это не экономия сама по себе, но:

- бюджет-гард `CLAUDE_DAILY_BUDGET_USD=1` срабатывает вдвое раньше, чем нужно
  (алёрты `enrich_submit_blocked_budget` ×2 в baseline — возможно, ложные блокировки выпуска);
- решения «дорого/недорого» принимаются по завышенной цифре.

Фикс: прокинуть `batch: true` в расчёт для batch-результатов (или хранить обе цифры:
list и billed-estimate). Потом осознанно перенастроить дневной бюджет.

### 2.6. Prompt caching — проверить, что он реально работает

`cache_control: ephemeral` на system prompt уже стоит. Но: в Batch API запросы обрабатываются
параллельно, и cache hit не гарантирован; плюс минимальный кешируемый префикс у Sonnet 4.6 —
2 048 токенов (наш system prompt примерно на границе). Дёшево проверить по
`llm_usage_logs.cache_read_tokens`: если по batch-вызовам стабильно ноль — кеш не работает,
и можно (а) для sync-вызовов reviewer-а убедиться, что system стабилен побайтово,
(б) не рассчитывать на кеш в экономике batch.

### 2.7. Ретраи валидации — чинить дёшево, а не переписывать дорого

`editorial_validation_failed` сейчас уходит на переотправку **новым батчем** (до 3 попыток,
каждая — полная генерация). Большинство провалов — мелочь: анкор не в первом предложении,
лишний `link_anchor`, длина teaser. Дешевле: перед ретраем прогонять «починочный» вызов
DeepSeek («вот JSON, вот список ошибок валидатора — поправь только их»), и только при
повторном провале — полный ретрай. Это копейки против полного прогона.

## 3. Надёжность = тоже деньги (урок инцидента 2026-06-09)

Кончились кредиты Anthropic → выпуск упал с 60–70 до 7 статей/день. То есть premium fallback —
**единая точка отказа выпуска**, хотя 90% статей пишет DeepSeek.

- **Degraded-режим:** при `credit balance too low` / 4xx от Anthropic не останавливать статью,
  а публиковать low-risk материалы по чистому DeepSeek-пути без reviewer-а (с пометкой в
  `llm_usage_logs`/`article_attempts`, чтобы потом догнать review-ом). High-risk — в
  `retry_wait` до восстановления. Канал и сайт продолжают жить.
- **Ранний алёрт по балансу:** критический алёрт не «retry-failed упал 3 раза», а «первый
  `credit balance too low`» — отдельный alert type с пушем владельцу сразу.
- (Зона владельца) auto-reload в консоли Anthropic, чтобы класс инцидента исчез.

## 4. Картинки

### 4.1. Дешевле: gpt-image-1-mini для рядовых обложек

Сейчас default — `gpt-image-1.5` low 1536×1024 = **$0.013**/шт. По той же ценовой таблице
(`pipeline/model-pricing.ts`): `gpt-image-1-mini` low 1536×1024 = **$0.006** (−54%),
а `gpt-image-2` low — $0.005, если доступ есть. Предложение: сгенерить тестовую пачку
10–15 обложек на mini по текущим scene-промптам, сравнить глазами с текущими. Если ок —
mini для рядовых, `gpt-image-1.5`/medium оставить только для homepage-priority (как сейчас).

### 4.2. Меньше платных генераций вообще

После cover mirroring + promotion-from-inline + `needsAiCover` через sanitizer спрос на
AI-обложки уже должен был упасть. Стоит снять цифру: сколько AI-обложек генерится в день
сейчас и для каких источников. Если какой-то источник стабильно требует AI-обложку — возможно,
дешевле улучшить извлечение картинок из него (селекторы fetcher), чем платить за генерацию.

### 4.3. Качество: прокидывать сущности статьи в промпт сцены

Scene matcher уже context-aware (Wave 4), но промпт всё ещё «сцена + категория». Дешёвое
улучшение без LLM: подставлять в промпт конкретику из статьи — название продукта/устройства,
тип объекта (chip/robot/glasses/app) из тех же regex-детекторов, что уже есть в scorer.
Обложка «hardware close-up of a pocket-sized device» бьёт «generic tech scene».

## 5. Качество текстов

### 5.1. Дешёвый ежедневный LLM-judge — метрика качества, которой сейчас нет

Сейчас качество ловится только validator-ом (структура) и глазами владельца. Предложение:
раз в день брать выборку 5–10 опубликованных статей и прогонять через Haiku-judge по рубрике
(фактическая опора на источник, якорь в лиде, запрещённые фразы, полезность контекста,
оценка 1–5). Стоимость — центы. Результат — строка в ops-report: средний балл по дню,
разбивка DeepSeek vs premium vs Haiku (если введём), топ-3 худших со ссылками.

Это даёт три вещи: (1) ранний сигнал деградации при смене моделей/промптов — без него
пункты 2.2–2.3 рискованно раскатывать; (2) данные «какой источник даёт слабые тексты»;
(3) объективную базу для решения «можно ли удешевлять дальше».

### 5.2. Год-санитайзер — расширить с captions на тела статей

`hasStaleYearHallucination` уже ловит «WWDC 2025» в Telegram-caption. Тот же дешёвый
детерминированный чек стоит включить в validator для `editorial_body`/`lead`/`ru_title`:
прошедший год допустим, только если он есть в исходнике. Это самый частый класс
фактических ляпов у дешёвых writer-моделей.

### 5.3. Банк удачных примеров в промпт DeepSeek (few-shot)

System prompt уже хороший, но без примеров. Один-два эталонных примера «исходник → итоговый
JSON» (выбрать из лучших опубликованных, сжать) обычно сильнее всего подтягивают слабые
модели по ритму и структуре. Цена — +1–2К токенов input по DeepSeek-тарифу с автокешем
(~$0.03 за MTok на cache hit) — пренебрежимо.

## 6. Отбор новостей

Здесь за последние недели сделано много (importance, story dedup, source caps), крупных дыр
не видно. Точечные идеи:

### 6.1. Замкнуть петлю обратной связи из Telegram

UTM в постах уже есть, но клики никуда не возвращаются. Дешёвый вариант без Метрики-API:
Telegram отдаёт **views по сообщениям канала** — забирать просмотры постов за вчера и писать
в `telegram_channel_posts.views`. Через 2–3 недели появится датасет «какие storyKey/источники/
категории реально смотрят» → можно аккуратно подмешать в `rankDigestCandidates` вес
«исторический интерес к этому типу сюжета» и в `lib/interest-ranking.ts`. (Цели Метрики на
CTA — отдельная зона владельца, уже зафиксировано в аудите.)

### 6.2. Адаптивный порог score для шумных источников

Daily cap (10/день на источник) лечит перекос объёма, но не качество хвоста: Habr-статья,
прошедшая 9-й в день, всё равно тратит enrichment. Идея: для источников с высоким
reject-rate / низким judge-score (см. 5.1) поднимать входной порог score на +1–2
конфигом в `feeds.config.ts`, вместо того чтобы платить за enrichment и потом отбраковывать.
Меньше мусора в пайплайн = меньше токенов = чище лента.

### 6.3. Мульти-источниковые сюжеты: один материал вместо трёх

`storyKey` уже умеет распознавать «один инфоповод из 3 источников» — сейчас это используется
только чтобы не дублировать в Telegram. Следующий уровень (дороже, на потом): при 3+
источниках на один strong storyKey собирать **один** сводный материал с несколькими
источниками вместо трёх отдельных страниц. Плюсы: нет каннибализации в поиске, материал
жирнее (SEO), enrichment-расход на сюжет падает втрое. Минусы: заметная переделка
pipeline-контракта — отдельная инициатива, не quick win.

## 7. Сводка: что делать в каком порядке

| # | Мера | Эффект на деньги | Риск для качества | Усилие |
|---|---|---|---|---|
| 1 | Фикс batch-скидки в учёте (2.5) | видимость ×2, чинит ложные блокировки бюджета | нет | S |
| 2 | Фикс ложных risk-флагов (2.1) | −30…50% premium-трафика (проверить замером) | нет | S |
| 3 | Кап входа premium-пути (2.4) | −30…50% стоимости Habr-лонгридов | минимальный | S |
| 4 | Degraded-режим без Anthropic (3) | защищает выпуск (инцидент №1 не повторится) | нет | M |
| 5 | high_score → balanced (2.2) | −80% на топ-статьях | низкий (reviewer остаётся) | S/M |
| 6 | LLM-judge выборка (5.1) | 0 (это страховка для 5 и 7) | — | M |
| 7 | Haiku для reviewer/fallback не-research (2.3) | −60…70% оставшегося premium | средний → судить по judge | M |
| 8 | gpt-image-1-mini (4.1) | −50% на картинках | проверить глазами | S |
| 9 | Год-санитайзер в body (5.2), repair-pass перед ретраем (2.7) | мелкая экономия + качество | нет | S |
| 10 | TG-views фидбек (6.1), адаптивный порог (6.2) | косвенный | нет | M |
| 11 | Сводные мульти-источниковые материалы (6.3) | −66% на повторных сюжетах | требует дизайна | L |

Грубая оценка итога по пунктам 1–8: расход на текст с ~$3–5/день к **~$0.5–1/день** при
том же объёме выпуска, картинки с $1 к ~$0.5/день. То есть месячный AI-бюджет дайджеста
с ~$120–180 к **~$30–45** — без снижения числа статей и с дополнительным контролем качества
(judge), которого сейчас нет.

Перед реализацией пунктов 2, 5, 7 обязательно: замер доли risk-флагов на реальных статьях
(скрипт-однодневка, read-only) и неделя judge-метрики как baseline.
