## 1. Архитектурное решение (один абзац)

Текущий поток: `RSS → Supabase (raw) → fetch HTML → DeepL (перевод заголовка + первых 2000 знаков) → Claude Haiku (один абстрактный why_it_matters) → Supabase (ru_title/ru_text/why_it_matters) → сайт (ru_text as-is) → TG (title + why_it_matters)`.

Новый поток: `RSS → Supabase (raw) → fetch полного HTML → Claude Sonnet (один структурированный вызов: русский заголовок + лид + тезисы + карточный тизер + TG-тизер + полный редакторский текст + quality_ok) → Supabase (новые поля) → сайт (lead крупно → summary буллетами → editorial_body абзацами) → TG (заголовок + tg_teaser + ссылка).`

Ключевые отличия:
- **DeepL из enrichment-пути удалён.** Claude сам переводит и перерабатывает — это один вызов и один стиль.
- **Редактор — Claude Sonnet 4.6, не Haiku.** Хайку для этого слабоват: даёт сжатый машинный тон.
- **Убирается ограничение в 2000 знаков.** Полная переработка может быть 4–8 тыс. знаков, если материал того стоит.
- **Все публичные поля генерируются в одном промпте** — consistency tone of voice.
- **Вводится жёсткий `quality_ok`** и публичные выборки читают только его.

### T1.4 Новый редакторский промпт

**Файл:** заменяем содержимое `pipeline/claude.ts`.

Интерфейс новой функции:

```ts
export interface EditorialOutput {
  ru_title: string
  lead: string
  summary: string[]
  card_teaser: string
  tg_teaser: string
  editorial_body: string
  quality_ok: boolean
  quality_reason: string   // если false — почему
}

export async function generateEditorial(
  originalTitle: string,
  originalText: string,
  sourceName: string,
  sourceLang: 'en' | 'ru',
  topics: string[],
): Promise<EditorialOutput | null>
```

Модель: `claude-sonnet-4-6`. Максимум выход ≈ 2500 токенов.

Системный промпт (дословно, копировать в код):

```
Ты — выпускающий редактор русскоязычного медиа Malakhov AI Дайджест.
Пишешь как спокойный деловой журналист. Референс тона — «Коммерсантъ», vc.ru.

ЖЁСТКИЕ ЗАПРЕТЫ. Никогда не используй:
- «важно отметить», «стоит обратить внимание», «в мире ИИ», «эта новость важна»,
  «как известно», «поистине», «впервые в истории», «это меняет всё», «прорыв»,
  «революция», «настоящий», «невероятный», «потрясающий».
- Канцеляризмы: «осуществлять», «в рамках», «что касается», «на сегодняшний день».
- Вводные пустышки: «что ж», «итак», «действительно».
- Маркетинговый пафос любого рода.

ПРАВИЛА.
- Пиши по-русски. «AI» = «ИИ». Названия моделей — в оригинале (GPT-4o, Claude 3.5 Sonnet,
  Gemini 2.5 Pro, Llama 3.1, YandexGPT 4, GigaChat).
- Английские термины оставляй только если нет русского аналога: LLM, API, open-source, benchmark.
- Первое предложение лида должно содержать минимум один конкретный якорь:
  имя собственное / число / дату / название продукта.
- Первый абзац никогда не повторяет заголовок.
- Никаких «Компания X объявила» в лиде — сразу суть.
- Ты можешь опираться ТОЛЬКО на факты из приведённого текста. Запрещено добавлять цифры,
  имена, продукты, которых в источнике нет.
- Если источник — Habr или vc.ru и текст содержит UI-мусор (строки вида «Уровень сложности»,
  «Время на прочтение», «Охват и читатели», «Читать далее», «Поделиться») —
  игнорируй их, они не являются частью статьи.

ЗАДАЧА. Получив оригинальный заголовок и текст, сгенерируй строго JSON со следующими полями:

{
  "ru_title": string,            // до 90 символов, русский, без кликбейта и без точки в конце
  "lead": string,                 // 1–2 предложения, 200–360 символов, с фактом в первой фразе
  "summary": string[],            // 3–5 буллетов, каждый 60–180 символов, по одному факту в каждом
  "card_teaser": string,          // 1 строка, 80–140 символов, чтобы кликнули на карточку
  "tg_teaser": string,            // 1–2 строки, 160–260 символов, чтобы кликнули из Telegram
  "editorial_body": string,       // основной текст 1500–5000 символов, 3–7 абзацев
  "quality_ok": boolean,          // true только если материал действительно пригоден к публикации
  "quality_reason": string        // если false — одно предложение «почему»; если true — пустая строка
}

СТРУКТУРА editorial_body.
Абзац 1: лид-развёртка. Что произошло, с конкретикой. Не повторяет поле lead дословно.
Абзац 2: контекст. Что этому предшествовало, почему это появилось сейчас.
Абзац 3: детали. Цифры, имена, цитаты, сравнение с предыдущим поколением.
Абзац 4: последствия. Кого это касается, на что влияет.
Абзацы 5–7 (опционально): расширение, если в источнике есть материал.

Разделяй абзацы одной пустой строкой (\n\n). Не используй маркдаун, списки, заголовки.

КРИТЕРИЙ quality_ok = true.
Все условия должны выполняться:
- В лиде есть минимум один конкретный якорь (имя собственное / число / дата / продукт).
- В editorial_body минимум 3 абзаца и минимум 800 символов.
- summary содержит минимум 3 пункта.
- Нет слов из списка запретов.
- Материал не повторяет заголовок в первом абзаце.
- В исходнике достаточно фактов, чтобы написать больше, чем пересказ заголовка.

Если хотя бы одно условие не выполнено — quality_ok = false и короткое объяснение в quality_reason.
Не пытайся «спасти» слабый материал пустыми абзацами — честно ставь false.

ВЫВОД. Только валидный JSON. Никаких пояснений до или после.
```

Пользовательское сообщение:

```
Источник: {sourceName}
Язык источника: {sourceLang}
Темы: {topics.join(", ")}

Оригинальный заголовок:
{originalTitle}

Оригинальный текст:
{originalText}
```

Параметры вызова:
- `model: 'claude-sonnet-4-6'`
- `max_tokens: 2500`
- `temperature: 0.4` (низкая, чтобы не уносило в сочинения)
- Поле `response_format` не задаём (Anthropic не принимает). Вместо этого парсим JSON вручную.

Парсинг ответа:
- Удалить возможные обёртки markdown ` ```json ... ``` `.
- `JSON.parse` → если падает, делать одну попытку `JSON.parse` после `.match(/\{[\s\S]*\}/)`.
- Если всё равно мусор — возвращать `null`, логировать, в базу **не писать**.

Валидация результата (строгая, иначе `null`):
- все поля присутствуют;
- `ru_title.length` 20..90;
- `lead.length` 100..400;
- `summary` массив, длина 3..5, каждый элемент 40..200;
- `card_teaser.length` 60..160;
- `tg_teaser.length` 120..300;
- `editorial_body.length ≥ 800`, `split("\n\n").length ≥ 3`;
- `quality_ok` boolean.

Если валидация провалилась — `null`, логируем причину, статья **не публикуется** (остаётся `enriched=true, published=false, quality_ok=false`, в `quality_reason` пишем причину).

**Приёмка T1.4.** На 10 тестовых статьях (разные источники) функция возвращает валидный `EditorialOutput` минимум для 7 из 10. Ручной просмотр 5 случайных: тон правильный, нет запретных слов, лид цепляет.

Новая структура:

```
Хлебные крошки

Мета (source badge, дата, темы, примерное время чтения)
```

- Заголовок — оставить `ru_title` (у нас он теперь редакторский, не машинный).
- У `featured` под заголовком — **показывать lead**, не card_teaser (больше места).

Также подчистить `truncate(why_it_matters, 120)` в `default` — уже не нужно, card_teaser изначально 80–140 символов.

**Приёмка T3.2.** Главная: заголовки понятные, под каждым — одна строка сути, а не «эта новость важна потому что». Выровнено, чисто.

### T3.3 Главная — заголовок-обещание

**Файл:** `app/page.tsx`.

Единственное «косметическое» изменение, которое сейчас нужно, потому что оно влияет на SEO и узнаваемость:
- H1 = `Malakhov AI Дайджест` + подзаголовок «Главное об искусственном интеллекте — {сегодняшняя дата на русском}».
- Заменить пустой заголовок «Последние новости» на этот.

Получение даты — на сервере через `Intl.DateTimeFormat('ru-RU', { day: 'numeric', month: 'long' })`. ISR `revalidate=300` у нас уже стоит.

**Приёмка T3.3.** Главная не выглядит как админка. В `<title>` стоит «Malakhov AI Дайджест», в `<h1>` видна дата.

### T3.4 Проверка всех ссылок

**Файл:** новый одноразовый `scripts/check-links.ts`.

Проходим по `getAllSlugs()`, делаем HEAD на `${NEXT_PUBLIC_SITE_URL}/articles/${slug}`, ожидаем 200. Если не 200 — логируем slug, проверяем руками.

Это страховка — у нас бот перед отправкой и так проверяет доступность (`isArticleLive` в `daily-digest.ts`), но раз в неделю полный прогон выявит битые статические страницы после ISR-миссов.

**Приёмка T3.4.** Скрипт выдаёт: `240/240 OK`. Если что-то не 200 — мы знаем раньше, чем заметит читатель.

## 5. Спринт 4 — Telegram-дайджест (1–2 дня)

### T4.1 Новый формат сообщения

**Файл:** `bot/daily-digest.ts`, функция `buildDigestText`.

Шаблон:

```
🤖 AI Дайджест · {день месяца на русском}
{N} главных событий в ИИ за сутки

{N}. {ru_title}
{tg_teaser}
→ {articleUrl}

...

Все {total_today} новостей дня: {siteUrl}
#AI {dynamic hashtags}
```

Изменения в коде:
- Убрать строку `Лучшие новости за день по версии редакции` — заменить на динамическую «5 главных событий в ИИ за сутки».
- Убрать эмодзи `💡` перед тизером — ничего не ставим, просто строка под заголовком.
- **Использовать `tg_teaser`, не `why_it_matters`**.
- Ссылка — отдельной строкой с символом `→`, без `[Читать на сайте →](url)` (лишняя markdown-обёртка).
- В конце — статистика «Все {total_today} новостей дня» с общей ссылкой.
- Добавить хэштеги: `#AI` всегда, плюс имена моделей/продуктов, распаршенные из `lead` по белому списку: `#OpenAI, #Anthropic, #Google, #YandexGPT, #GigaChat, #Claude, #GPT, #Gemini, #Llama, #Mistral, #HuggingFace`. Если в лиде найдены эти слова — добавить хэштеги (уникально, до 5).

Превью ссылки: включить **только для первой ссылки** (`disable_web_page_preview: false` для всего сообщения, либо разбить на два сообщения — это усложнение, пока оставим один пост с общим превью первой ссылки).

### T4.2 UTM-метки и слуги

**Файл:** `bot/daily-digest.ts`.

Каждый URL в TG получает:
```
?utm_source=tg&utm_medium=digest&utm_campaign=daily_{YYYYMMDD}&utm_content={position}
```

где `position` — номер в дайджесте (1..5).

Это нужно для Яндекс.Метрики (даже если счётчик ещё не стоит, метки уже в ссылках).

### T4.3 Выборка кандидатов — жёстче

**Файл:** `bot/daily-digest.ts`, запрос к Supabase.

Сейчас:
```ts
.eq('published', true)
.eq('tg_sent', false)
.order('score', { ascending: false })
.limit(5)
```

Новое:
```ts
.eq('published', true)
.eq('quality_ok', true)                 // критично
.eq('tg_sent', false)
.not('tg_teaser', 'is', null)           // страховка
.not('slug', 'is', null)
.gte('pub_date', from.toISOString())
.lte('pub_date', to.toISOString())
.order('score', { ascending: false })
.order('pub_date', { ascending: false })
.limit(8)                                // берём запас
```

Потом в коде: после `filterLiveArticles` (проверка 200 на сайте) оставить **не более 5**. Если после фильтра меньше 3 — отправить `Health-отчёт` в админ-чат с темой «Сегодня в дайджесте всего {N} статей» и не отправлять в канал (лучше пропустить день, чем слабый дайджест).

Отдельно: total_today считаем отдельным запросом — `published=true AND quality_ok=true AND pub_date BETWEEN from AND to`.

### T4.4 Поломка markdown — проверка

**Файл:** `bot/daily-digest.ts`, функция `escapeMd`.

Сейчас экранируется `_ \` [`. После нового формата:
- Заголовки обёрнуты в `**` — символ `*` в `ru_title` (редко, но возможно) поломает жирный. Добавить в escape: `*`.
- `tg_teaser` идёт без обёртки — в нём могут быть `_` и `*` → убирать при сборке поста.
- Ссылки теперь без markdown-обёртки `[...](...)`, а сырые — значит `[` и `]` в `tg_teaser` не страшны, но `<>` Telegram интерпретирует как HTML. Мы используем `parse_mode: 'Markdown'`. Оставляем escape `_ * \` [ ]`.

Тест: отправить на себя (личный чат) пост с разными спецсимволами. Убедиться, что парсинг не падает (`Telegram API вернул ok: false`).

### T4.5 Прогон и приёмка

**Ручной запуск:** `FORCE_DIGEST=1 npm run tg-digest`.

**Приёмка T4.x в целом.** В тестовом канале (или в личке админа) видишь:
- Шапка с датой.
- 3–5 пунктов, каждый: заголовок, 1–2 строки сути, ссылка.
- Первая ссылка имеет превью (Telegram рендерит OG-карточку статьи).
- В конце — строка со статистикой и хэштеги.
- Кликаешь на ссылку — открывается статья на сайте, которая читается.

## 6. Спринт 5 — Обкатка и приёмка (3 дня, параллельно с обычной жизнью)

### T5.1 Ручной просмотр 20 последних статей

Взять список из `getLatestArticles(20)`, прочитать каждую.

Для каждой статьи ответить **да/нет** на три вопроса:
1. Читается ли lead с интересом?
2. Понял ли я суть только по editorial_body (без перехода в оригинал)?
3. Стал бы я делиться этой ссылкой?

**Приёмка:** минимум **16 из 20 = «да» по всем трём**. Иначе — возвращаемся к T1.4, правим промпт, снова прогоняем T1.6.

### T5.2 Ручной просмотр 3 TG-дайджестов подряд

Три дня подряд выходит новый формат. Критерии:
- Ни одного пункта с пустым tg_teaser.
- Первая ссылка даёт превью.
- Статистика «Все N новостей» правдива (сверить с Supabase).
- Ни одного сломанного markdown’а.

### T5.3 Проверка ссылок

Запустить `npx tsx scripts/check-links.ts`. Все 200. Если нет — разбор полётов.

### T5.4 Мониторинг расходов Anthropic

Открыть Anthropic Console, смотреть дневной расход. Ожидание: $0.30–0.80/день при 20–40 статьях. Если сильно больше — в промпте слишком много входа, обрезать `MAX_TEXT_LENGTH` в fetcher с 8000 до 5000.

## 9. Контрольный список для запуска одного спринта

При старте каждого спринта:
1. Создать ветку от `main`: `feature/content-sprint-<N>`.
2. Сделать все T*.* из спринта.
3. Локально: `npm run build` — зелёный.
4. Локально: `npx tsx pipeline/enricher.ts` на тестовых данных (если спринт это трогает).
5. Открыть Preview на Vercel, проверить 3–5 страниц глазами.
6. PR в `main`. Merge.
7. В следующие сутки — убедиться, что GH Actions `enrich.yml` и `tg-digest.yml` отработали.
