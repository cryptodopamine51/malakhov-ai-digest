# Спека: медиа-стратегия для статей без API-генерации и расхода токенов

## Проблема

Сейчас изображения и схемы в статье могут тупо падать в конец страницы отдельным блоком, без связи с ходом текста. Это выглядит как dump ассетов, а не редакционная работа.

## Цель

Перейти от хвостового gallery-dump к встроенным, уместным и дешёвым визуалам внутри статьи:
- только там, где они действительно помогают пониманию;
- без вызовов внешних image API;
- без расхода LLM-токенов на генерацию картинок.

## Принципы

- Не вставлять “просто любую картинку”.
- Не публиковать чужие изображения без явной причины и понятной роли.
- Предпочитать:
  - таблицы,
  - сравнительные блоки,
  - простые SVG-схемы,
  - ASCII-like or CSS-native diagrams,
  - редакционные callout-блоки,
  - timeline / matrix / metrics cards.

## Целевая модель

### 1. Inline media slots внутри editorial_body

Нужен механизм, позволяющий вставлять визуал не в конец статьи, а между абзацами.

Возможные варианты реализации:

1. Structured body schema:
   - `article_blocks: Array<{ type: 'paragraph' | 'table' | 'image' | 'diagram' | 'quote' | 'glossary-inline', ... }>`
2. Lightweight markup inside `editorial_body`:
   - специальные токены вроде `[[diagram:comparison-models]]`
3. Hybrid:
   - `editorial_body` остаётся строкой;
   - дополнительные `inline_media` описывают якоря вставки.

**Рекомендация:** не пытаться хакать картинки регексом по готовому plain text. Лучше перейти на structured blocks или хотя бы на anchor-based inline slots.

### 2. Типы поддерживаемых визуалов

На первой итерации:
- `table`
- `comparison`
- `timeline`
- `metric-cards`
- `simple-svg-diagram`
- `source-image` только если:
  - картинка напрямую относится к сюжету;
  - не является UI-мусором;
  - у неё есть понятный caption.

### 3. Правила отбора исходных картинок

- Не брать все `article_images` из источника подряд.
- Отбрасывать:
  - декоративные иконки;
  - баннеры;
  - логотипы без смысловой нагрузки;
  - screenshots интерфейса без контекстной ценности;
  - схемы низкого качества или нерелевантные тексту.

Нужен heuristic filter на уровне пайплайна:
- размер / aspect ratio если доступно;
- alt/caption keywords;
- позиция в DOM;
- исключение повторяющихся asset-host patterns.

## Что сделать

### Этап A. Быстрый cleanup текущей модели

- Убрать автоматический dump `article_images` внизу статьи как дефолтное поведение.
- Сохранить только те медиа, которые могут быть встроены осмысленно.

### Этап B. Спроектировать схему данных

Варианты новых полей:
- `article_blocks JSONB`
- или `inline_media JSONB`
- или `article_visuals JSONB`

Нужно описать:
- формат;
- backward compatibility;
- миграцию старых статей.

### Этап C. Рендер

- Рендерер статьи должен поддерживать визуалы между абзацами.
- Для таблиц и схем использовать design tokens текущего editorial UI.
- SVG-схемы должны собираться кодом локально, без внешних API.

## Тесты

### Unit

- Парсинг structured blocks / inline media schema.
- Filter logic для source images.
- Рендер таблицы / comparison block / simple diagram.

### Snapshot / component

- Одна статья с inline table.
- Одна статья с inline SVG diagram.
- Одна статья без визуалов.

### Manual

- Открыть минимум 5 статей с медиа.
- Убедиться, что:
  - визуал стоит внутри логики повествования;
  - внизу нет dump-галереи;
  - caption объясняет роль изображения;
  - mobile layout не ломается.

## Критерии приёмки

- Картинки и схемы больше не сбрасываются тупо в конец статьи по умолчанию.
- Хотя бы один поддерживаемый inline visual type работает end-to-end.
- Архитектура не зависит от image API и не расходует токены на генерацию графики.
