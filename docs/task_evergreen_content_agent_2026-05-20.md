# Task: Autonomous Evergreen Content Agent

Дата: 2026-05-20

Проект: Malakhov AI Digest (`news.malakhovai.ru`)

## Что нужно сделать простыми словами

Нужно собрать локального контент-агента, который помогает системно выпускать evergreen-статьи для сайта: не просто генерирует текст, а проходит весь редакционный путь от темы до готового пакета для публикации.

Агент должен брать тему из списка, понимать поисковый интент, смотреть конкурентов и источники, проверять, нет ли у нас уже похожей статьи, писать сильный черновик, редактировать его до уровня публикации, готовить SEO-метаданные, FAQ, внутренние ссылки, ТЗ на картинки и финальный пакет для Codex.

Главная цель: статьи должны собирать органический трафик, усиливать тематические кластеры сайта и читаться как нормальные полезные материалы для предпринимателя, руководителя, маркетолога или продакта, а не как шаблонный AI-текст.

Работать будем так:

1. Сначала фиксируем список 30 тем как редакционный backlog.
2. Для каждой темы агент создаёт отдельную рабочую папку с SEO-брифом, исследованием, черновиком, редактурой, источниками, картинками и чеклистом.
3. Готовая статья попадает в текущую evergreen-архитектуру сайта: `content/guides/<slug>.md`, metadata/FAQ/related links, локальные изображения в `public/images/guides/<slug>/`.
4. Картинки не генерируем через платный API. Агент готовит точные промпты, alt и подписи, а изображения создаём через ChatGPT/Codex-подписку или локальные SVG/схемы, затем кладём в проект как WebP.
5. Перед публикацией Codex проверяет сборку, мобильную версию, ссылки, картинки, schema и соответствие SEO-стандарту.

## Цели

- Выпустить первые 30 evergreen-материалов по ключевым AI-темам.
- Построить SEO-кластеры вокруг ИИ для бизнеса, ИИ-агентов, маркетинга, AI coding и инструментов.
- Сделать процесс повторяемым: одна тема должна превращаться в готовый publication package без хаотичных ручных шагов.
- Сохранить редакционный угол Malakhov AI: практическое внедрение, бизнес-эффект, ограничения, стоимость, риски, реальные сценарии и выводы для руководителя.
- Подготовить контент так, чтобы Codex мог безопасно разложить материалы по проекту и не ломать текущую архитектуру.

## Текущая архитектура публикации

Evergreen-гайды сейчас живут отдельно от RSS/news pipeline:

- текст: `content/guides/<slug>.md`;
- metadata, FAQ, cover, inline images, related links: `lib/guides.ts`;
- страница: `app/guides/[slug]/page.tsx`;
- список гайдов: `app/guides/page.tsx`;
- картинки: `public/images/guides/<slug>/`;
- canonical URL: `https://news.malakhovai.ru/guides/<slug>`;
- schema: `Article`, `FAQPage` только если FAQ виден на странице, `BreadcrumbList`.

Первый материал уже существует:

- тема: `Как внедрить ИИ в бизнес в 2026 году`;
- slug: `kak-vnedrit-ii-v-biznes-2026`;
- текст: `content/guides/kak-vnedrit-ii-v-biznes-2026.md`;
- metadata: `lib/guides.ts`;
- изображения: `public/images/guides/kak-vnedrit-ii-v-biznes-2026/`.

Правило: агент не должен создавать дубль для уже существующего интента. Если тема совпадает с существующим гайдом, он готовит update package.

## Предлагаемый формат topic backlog

Чтобы агенту было удобно работать, список тем нужно вынести в структурированный файл:

`content/evergreen/topics.json`

Рекомендуемые поля:

```json
{
  "id": 2,
  "status": "planned",
  "title": "Сколько стоит внедрение ИИ в компанию",
  "cluster": "ИИ для бизнеса",
  "primaryKeyword": "сколько стоит внедрение ИИ",
  "supportingKeywords": [
    "стоимость внедрения ИИ в бизнес",
    "цена AI автоматизации",
    "сколько стоит нейросеть для компании"
  ],
  "intent": "practical",
  "audience": [
    "предприниматель",
    "собственник",
    "CEO",
    "COO"
  ],
  "targetSlug": "skolko-stoit-vnedrenie-ii-v-kompaniyu",
  "priority": "high",
  "cta": "architectural_ai_audit",
  "notes": "Сделать сильный калькулятор бюджета: пилот, интеграции, данные, поддержка, API, ошибки."
}
```

Статусы:

- `planned` - тема есть, работа не начата;
- `researching` - собирается SEO-исследование;
- `drafted` - есть черновик;
- `editing` - идёт финальная редактура;
- `ready_for_codex` - готов publication package;
- `published` - материал добавлен на сайт;
- `needs_update` - нужно обновить существующую статью;
- `blocked` - не хватает данных, решения или источников.

## Процесс агента

### 1. Topic intake

Вход:

- тема;
- кластер;
- аудитория;
- предполагаемый slug;
- CTA;
- дополнительные заметки владельца.

Выход:

- `00-topic.json`;
- решение: новая статья, обновление существующей статьи или блокировка.

### 2. Anti-cannibalization check

Проверить:

- `content/guides/`;
- `lib/guides.ts`;
- текущие sitemap/live URL, если доступно;
- близкие статьи в категориях;
- будущие темы из `content/evergreen/topics.json`.

Выход:

- список похожих URL/тем;
- решение `create | update | merge | defer`;
- какие внутренние ссылки нужны между материалами.

### 3. SEO brief

Собрать:

- основной запрос;
- 3-8 дополнительных запросов;
- поисковый интент;
- уровень осведомлённости аудитории;
- цель статьи;
- место в структуре сайта;
- title;
- description;
- H1;
- target slug;
- planned internal links;
- CTA;
- required image assets;
- fact boundaries.

Выход: `01-seo-brief.md`.

### 4. SERP and reference research

Для evergreen/manual статьи нужно изучить 5-10 сильных материалов:

- какие блоки повторяются;
- какие вопросы закрывают;
- где слабые места;
- чего не хватает предпринимателю/руководителю;
- какой уникальный угол даст Malakhov AI.

Важно:

- референсы не переписывать;
- использовать их для интента, структуры и глубины;
- факты, цифры, цены, юридические и продуктовые утверждения проверять по первичным/надёжным источникам.

Выход: `02-serp-research.md` и `03-source-notes.md`.

### 5. Outline

Структура должна включать:

- H1;
- краткое резюме;
- оглавление;
- H2/H3;
- таблицы;
- чеклисты;
- практические сценарии;
- ограничения;
- ошибки;
- риски;
- блок `Вывод Malakhov AI`;
- FAQ;
- CTA;
- внутреннюю перелинковку;
- места для изображений.

Выход: `04-outline.md`.

### 6. Draft

Черновик пишет роль из Проекта 1:

- экспертно;
- плотно;
- без воды;
- с примерами;
- с таблицами;
- с локальным контекстом РФ/СНГ, где уместно;
- без нейросетевой гладкости;
- без неподтверждённых цифр.

Ориентиры объёма:

- главные хабы: 10 000-18 000 знаков;
- статьи про инструменты: 6 000-12 000 знаков;
- глоссарные/узкие объяснения: 2 000-5 000 знаков.

Выход: `05-draft.md`.

### 7. Editorial pass

Финальную редактуру делает роль из Проекта 2:

- проверить тезис;
- усилить практическую пользу;
- убрать повторы и общие места;
- проверить структуру H2/H3;
- проверить SEO;
- добавить snippet-блоки;
- уточнить FAQ;
- подготовить финальную статью;
- подготовить ТЗ для Codex.

Выход: `06-editorial-pass.md` и `07-final-article.md`.

### 8. Fact and source pass

Для важных claims нужен отдельный список:

```text
Утверждение | Источник | Дата проверки | Риск | Комментарий редактора
```

Обязательно проверять:

- цены;
- даты;
- названия продуктов;
- ограничения тарифов;
- юридические/регуляторные утверждения;
- российский рынок;
- сравнения инструментов;
- производительность моделей;
- цитаты.

Если факт нестабилен, в статье нужно писать дату контекста или формулировать осторожно.

### 9. Image brief

Для каждой статьи подготовить:

- cover prompt;
- 2-4 inline image prompts или SVG-схемы;
- alt;
- caption;
- место вставки после конкретного H2;
- размер;
- filename;
- negative prompt / anti-cliche notes.

Правило для этого проекта:

- не использовать image API (политика проекта — `docs/spec_2026-05-21_evergreen-quality-standard.md`, §4);
- для сложных иллюстраций использовать подписку ChatGPT (Plus / Pro / Codex);
- для схем, матриц, roadmaps и таблиц по возможности делать локальные SVG/HTML/CSS и экспортировать в WebP;
- обложка evergreen-гайда: 1200×675 WebP, ≥ 80 KB (тонкий SVG-WebP — недостаточно для cover);
- inline images: обычно 1200×800 WebP (или 1200×1200 для квадратов);
- в изображениях избегать читаемого текста, роботов, светящихся мозгов, неона, рукопожатия человека и робота, generic office stock.

Workflow:

1. Codex/агент готовит `09-image-brief.md` с `prompt`, `negative_prompt`, `alt`, `caption`, `aspect`, `filename_png`, `filename_webp` для cover и каждой inline-картинки.
2. Владелец/редактор открывает ChatGPT, генерит PNG, сохраняет в `content/evergreen/packages/<slug>/raw-images/<filename>.png` (имя совпадает с brief).
3. `npm run images:prep -- --slug=<slug>` (`scripts/images-prep.ts`) ресайзит и конвертит PNG в WebP (cover 1200×675, inline rect 1200×800, square 1200×1200, quality 82) и кладёт в `public/images/guides/<slug>/<filename>.webp`. PNG > 5 МБ → warn.
4. `npm run evergreen:check -- --slug=<slug>` проверяет наличие cover ≥ 80 KB и inline-файлов.

SLA владельца на cover после `ready_for_codex`: 48 часов; иначе гайд переходит в `blocked: cover_pending`.

Выход: `09-image-brief.md` и готовые файлы в `public/images/guides/<slug>/`.

Выход: `11-publication-checklist.md`.

## Требования к качеству статьи

Каждая статья должна отвечать на 5 вопросов:

1. Что читатель понял за первые 20 секунд?
2. Какую практическую задачу он сможет решить после чтения?
3. Где в тексте есть конкретика: цифры, сценарии, таблицы, чеклисты, примеры?
4. Что в статье есть такого, чего нет в обычной выдаче?
5. Какой следующий шаг на сайте логично сделать после чтения?

Запрещено:

- писать общие советы без механики;
- переписывать конкурентов;
- выдумывать цены, кейсы, числа и продуктовые функции;
- использовать инфобизнес-клише;
- раздувать текст ради объёма;
- делать FAQ, которого нет в видимом тексте;
- ставить внутренние ссылки на будущие страницы как будто они уже опубликованы.

## Кластерная стратегия

### ИИ для бизнеса

Опорный хаб уже начат статьёй `Как внедрить ИИ в бизнес в 2026 году`.

Остальные статьи должны углублять отдельные интенты:

- стоимость;
- малый бизнес;
- процессы;
- ошибки;
- выбор первого проекта.

Важно избежать каннибализации: эти статьи должны ссылаться на главный гайд, но не повторять его целиком.

### ИИ-агенты

Сделать отдельный подкластер:

- базовое объяснение;
- отличие от чат-бота;
- продажи;
- поддержка;
- документы;
- архитектура.

Нужен единый словарь: агент, чат-бот, workflow, tool use, memory, retrieval, human-in-the-loop, logs, permissions.

### Маркетинг и контент

Угол: не "нейросети пишут посты", а production-система для маркетинга:

- сценарии;
- контент-завод;
- SMM;
- Яндекс.Директ;
- изображения;
- видео.

Нужно больше практических таблиц, workflow и ограничений по качеству/бренду/правам.

### AI coding / vibe coding

Писать для предпринимателя и продуктовой команды, а не только для разработчика:

- что можно собрать;
- где риски;
- как ставить задачи;
- как контролировать качество;
- где Codex/Claude Code/Replit реально помогают;
- где нужен инженер.

### Инструменты и сравнения

Самый высокий риск устаревания. Нужны даты проверки и осторожные формулировки:

- тарифы и функции быстро меняются;
- сравнения должны обновляться чаще;
- каждая статья должна иметь блок `Актуальность проверена: <дата>`;
- для ChatGPT/Claude/DeepSeek/Perplexity/n8n/Make нужны официальные источники там, где речь о функциях, тарифах и ограничениях.

## Технические задачи перед массовым выпуском

### 1. Сделать structured topic backlog

Создать `content/evergreen/topics.json` с 30 темами, slug, cluster, intent, priority, CTA и status.

### 2. Развязать `lib/guides.ts` от одного гайда

Сейчас metadata жёстко записана в одном файле. Для 30 материалов лучше перейти к одному из вариантов:

- `content/guides/<slug>.meta.json` рядом с markdown;
- `content/guides/meta/<slug>.json`;
- общий `content/guides/index.json`.

После этого `lib/guides.ts` должен читать registry и markdown, а не хранить весь список вручную.

### 3. Добавить content package validator

Нужен локальный скрипт:

```bash
npm run evergreen:check -- --slug=<slug>
```

Он должен проверять:

- есть ли markdown;
- есть ли metadata;
- есть ли cover;
- совпадает ли FAQ в metadata и тексте;
- есть ли alt/caption у картинок;
- нет ли ссылок на несуществующие internal URLs;
- slug ASCII;
- title/description не пустые;
- related links существуют.

### 4. Добавить удобный scaffold

Нужен локальный скрипт:

```bash
npm run evergreen:new -- --topic-id=2
```

Он создаёт рабочую папку пакета, копирует шаблоны и подставляет тему.

### 5. Обновить guide render при необходимости

Проверить:

- CTA не должен быть захардкожен под один гайд;
- inline image insertion должен работать для разных статей;
- related links должны поддерживать 2-5 ссылок;
- таблицы должны оставаться удобными на мобильном;
- `FAQPage` должен всегда соответствовать видимому FAQ.

## Definition of Done для агента

Агент считается готовым для пилота, если он может по одной теме:

- создать рабочий пакет;
- сделать SEO-бриф;
- провести anti-cannibalization check;
- собрать исследование и источники;
- подготовить структуру;
- написать черновик;
- отредактировать финальную статью;
- подготовить metadata/FAQ/related links;
- подготовить image brief без API;
- подготовить task для Codex;
- пройти checklist публикации.

Агент считается готовым для серии из 30 статей, если дополнительно:

- есть `topics.json`;
- есть шаблоны;
- есть validator;
- guide registry не требует ручного копирования больших блоков в `lib/guides.ts`;
- процесс поддерживает `create` и `update`;
- есть журнал статусов по каждой теме;
- можно понять, какие статьи опубликованы, какие в редактуре, какие заблокированы.

## Открытые вопросы

1. Подтвердить финальный формат хранения topic backlog: JSON или CSV.
2. Подтвердить, все 30 материалов публикуются как `/guides/<slug>`.
3. Решить, нужен ли отдельный публичный хаб `/guides` с фильтрами по кластерам.
4. Зафиксировать CTA по кластерам: Telegram, AI-разбор, чеклист, консультация, другой лид-магнит.
5. Решить, кто указан как author в schema: организация или Иван Малахов.
6. Решить, сколько картинок минимум нужно для обычной статьи: cover + 2 inline или cover + 4 inline.
7. Решить, делаем ли схемы локально как SVG, а концептуальные обложки через ChatGPT/Codex.
8. Подтвердить, нужны ли отдельные update cadences: инструменты ежемесячно, бизнес-гайды раз в квартал.
9. Решить, нужны ли лид-магниты под статьи: чеклист внедрения, калькулятор ROI, матрица выбора проекта.
10. Решить, какие внешние источники считать допустимыми для тарифов, функций и продуктовых сравнений.
