# Карта Синхронизации Документов И Задач

## Назначение

Документ нужен, чтобы план, задачи и критерии качества не расходились между собой.

Связанные документы:

- [Основной продуктовый план](/Users/malast/malakhov-ai-digest/docs/newsroom_product_plan.md)
- [Бэклог исполнения](/Users/malast/malakhov-ai-digest/docs/newsroom_execution_backlog.md)
- [Матрица тестов и приемки](/Users/malast/malakhov-ai-digest/docs/newsroom_quality_matrix.md)

## Роли Документов

### `newsroom_product_plan.md`

Содержит:

- общую цель
- продуктовые принципы
- крупные фазы
- целевое состояние

Не должен содержать:

- мелкий технический бэклог
- детальные тест-кейсы по каждому файлу

### `newsroom_execution_backlog.md`

Содержит:

- конкретные задачи
- порядок исполнения
- зависимости
- точки изменения
- критерии приемки по задачам

Не должен содержать:

- размытые продуктовые лозунги вместо задач

### `newsroom_quality_matrix.md`

Содержит:

- автоматические проверки
- ручную приемку
- запреты для публичного слоя

Не должен содержать:

- описание архитектуры
- список будущих идей без критериев

## Правила Синхронизации

1. Если добавляется новая крупная фаза в основном плане, в бэклоге должна появиться минимум одна задача на исполнение этой фазы.

2. Если в бэклоге появляется новая задача, в матрице качества должны появиться проверки, которые подтверждают ее завершение.

3. Если меняется визуальное направление фронтенда, это должно обновляться:

- в основном плане
- в задачах фронтенда
- в ручной приемке качества

4. Если меняются правила публичного русского языка, это должно обновляться:

- в основном плане
- в задачах редакционного слоя
- в автоматических тестах

5. Если внедряется новый публичный слой, например изображения или редакторская ручная правка, он должен быть отражен:

- в основном плане как фаза или подфаза
- в бэклоге как конкретные задачи
- в матрице качества как приемка

## Проверка Перед Каждой Новой Итерацией

Перед началом новой итерации нужно проверить:

1. Какой пункт основного плана сейчас в работе.
2. Какие задачи из бэклога относятся к этому пункту.
3. Какие тесты и критерии приемки подтверждают завершение.
4. Не появились ли новые требования, которые надо добавить сразу во все документы.

## Проверка После Каждой Новой Итерации

После завершения итерации нужно проверить:

1. Изменился ли объем продукта.
2. Нужны ли новые задачи в бэклоге.
3. Нужны ли новые тесты и приемка.
4. Не противоречат ли документы текущему коду и состоянию продукта.

## Текущее Соответствие

На текущий момент документы покрывают:

- спасение публичной витрины
- редакционный порог публикации
- редакционный русский язык
- переработку статей под наш стиль
- фронтенд-переупаковку
- поисковую оптимизацию
- изображения
- ручной редакторский контур

## Что Считать Рассинхронизацией

Рассинхронизация есть, если:

- задача уже делается в коде, но не описана в бэклоге
- задача есть в плане, но для нее нет критериев приемки
- тесты проверяют одно, а план требует другое
- фронтенд-референс не отражен в задачах на дизайн и верстку
