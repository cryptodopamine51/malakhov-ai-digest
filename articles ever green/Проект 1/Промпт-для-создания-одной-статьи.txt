Создай вечнозелёную SEO-статью для news.malakhovai.ru.

Тема: [ВСТАВИТЬ ТЕМУ]
Целевой кластер: [например: ИИ для бизнеса / ИИ-агенты / AI coding / нейросети для маркетинга]
Целевая аудитория: предприниматели, руководители, маркетологи, продакты, владельцы малого и среднего бизнеса.
Цель статьи: собрать органический трафик и привести читателя к подписке, заявке на AI-разбор или дальнейшему чтению на сайте.

Сначала сделай SEO-бриф:
— основной поисковый запрос;
— дополнительные запросы;
— поисковый интент;
— предполагаемый URL;
— title;
— description;
— H1;
— место статьи в структуре сайта;
— какие внутренние страницы сайта должны на неё ссылаться;
— на какие страницы она должна ссылаться.

Затем проанализируй 5–10 сильных материалов по теме:
— какие блоки у них повторяются;
— какие вопросы они закрывают;
— чего им не хватает;
— как сделать нашу статью глубже и полезнее;
— какой уникальный угол должен быть у Malakhov AI.

Не переписывай чужие тексты. Используй их только как референсы для анализа интента и структуры.

После анализа напиши полную статью:
— экспертно;
— с практическими примерами;
— с таблицами;
— с чеклистом;
— с блоком рисков;
— с блоком «Вывод Malakhov AI»;
— с FAQ;
— с рекомендациями для изображений;
— с CTA.

Структура статьи должна быть готова для публикации на сайте.

В конце добавь:
1. SEO-пакет.
2. Список источников.
3. Рекомендации по внутренней перелинковке.
4. Идеи для 3 связанных статей.
5. Что проверить редактору перед публикацией.

================================================================
Обязательные элементы любой статьи (Evergreen Quality Bar 2026-05-21)
================================================================

В каждой статье обязательно:

1. Лид с factual anchor в первом предложении: число с единицей, имя собственное, дата или источник.
2. Видимая дата проверки актуальности — поле `verifiedAt` в metadata; в шапке статьи render «Актуальность проверена: <дата>».
3. Минимум один развёрнутый numerical worked example для статей с числовым intent (стоимость, ROI, окупаемость, метрики, сравнения тарифов). Шаблон: «Ситуация → данные → формула → результат → выводы». Не подменять расчёт таблицей категорий расходов.
4. Минимум один развёрнутый кейс по структуре:

   ### <короткий заголовок кейса>
   В опубликованном тексте не добавляй служебный блок «Источник: редакционный пример» и не ставь пометку «Редакционный пример». Если кейс не публичный и не клиентский, оформляй его нейтрально как обезличенный сценарий без заявления «наш кейс».

   Ситуация. <2–3 предложения с конкретикой: масштаб, отрасль, проблема>
   Что делает ИИ. <процесс>
   Что нужно для пилота. <чек-лист>
   Метрики. <2–4 KPI с цифрами>
   Бизнес-логика / итог. <вывод>

   Иерархия источников: реальный публичный кейс (McKinsey/BCG/Gartner/Habr/vc.ru/«Яков и Партнёры»/IDC/НИУ ВШЭ) → анонимизированный («компания из e-commerce, ~1000 заявок/день») → редакционный сценарий без видимой служебной пометки.

5. Counter-strategy блок «Когда не стоит / не окупится / не подходит / какие сценарии пропустить» — 3–5 практических критериев. Не «когда нет бюджета», а конкретно: «нет повторяемости процесса», «нет владельца результата», «нет SLA», «процесс меняется чаще раза в месяц».
6. Российский контекст и регуляторика:
   — для статей про данные/клиентов/HR/договоры — обязательно упомянуть 152-ФЗ, режим коммерческой тайны;
   — для тарифов — GigaChat / YandexGPT как локальные альтернативы;
   — для маркетинга — Яндекс.Директ, ВКонтакте, OK как локальные surfaces.
7. ≥ 2 внутренних ссылок в теле статьи на `/guides`, `/categories/<category>` или `/russia` (не считая related в footer). Linking на ещё не опубликованные гайды запрещено.
8. CTA cap: не более 2 inline-CTA в теле + 1 final-CTA блок с 3 карточками. CTA указывают ТОЛЬКО на реально существующие поверхности проекта:
   — `telegram-digest` → `@malakhovaidigest` (ежедневный AI-дайджест в Telegram, реально выходит).
   — `contacts` → `malakhovai.ru/contacts` (форма заявки на консультацию, реально работает).
   — `telegram-personal` → `@iddopamine` (личный Telegram).
   Запрещено обещать чеклисты, PDF, гайды на почту, бесплатные шаблоны и любые lead-magnet артефакты, которых физически нет. Дефолтный final-CTA набор — `DEFAULT_FINAL_CTA_CARDS` в `app/guides/[slug]/page.tsx`. Override через `inlineCtas`/`ctaCards` в meta — допустим, но интент и action должны указывать на одну из трёх поверхностей выше.
9. Метаданные включают `verifiedAt` (ISO-date), `caseSourcing` (`public` / `anonymized` / `editorial` — опционально). Cover и inline images перечислены с alt и caption.
10. Картинки готовятся ТОЛЬКО через подписку ChatGPT (Plus/Pro/Codex). Image API любого вендора не используется. Cover 1200×675 WebP (quality 88), inline 1200×800 (или 1200×1200 квадрат, quality 88), effort 6, smartSubsample=false (полное 4:4:4 chroma). `npm run images:prep` также создаёт `-480.webp` и `-768.webp` variants. Локальные SVG-диаграммы допустимы для матриц/roadmap/калькуляторов.

   **SEO-convention для filename'ов** (отражается в `08-metadata.json::cover.src` и `inlineImagesByHeading[*].src` ДО генерации PNG):
   — Cover: `<slug>-cover.webp` (или `<primary-keyword>-<short-modifier>.webp` если slug длиннее ~40 символов).
   — Inline: `<slug-short>-<section-keyword>.webp`, где slug-short — первые 2–4 значимые слова slug'а гайда, а section-keyword — короткое описание содержимого (`scenarii`, `plan-30-dney`, `matrica-vybora`, `kogda-ne-stoit`).
   — ASCII only, lowercase, hyphens, ≤ 60 символов. Не использовать generic-имена `image1.webp`, `diagram.webp`, `cover.webp` — они теряют SEO-сигнал.
   — Владелец может класть PNG в `raw-images/` с любым именем (ChatGPT часто отдаёт `ChatGPT_image_<ts>.png`); `npm run images:prep` автоматически переименует выход по SEO convention из meta. Pass 1 — точный stem-матч; pass 2 — random-имена маппятся по алфавитному порядку на declared meta order (cover → inline в порядке `inlineImagesByHeading`).
11. Worked example — только статический Markdown (текст + таблица + список). Никаких React Client Components / интерактивных калькуляторов на странице гайда (owner decision 2026-05-22).
12. Российские кейсы и цифры берутся из внутреннего white-list 8 источников (см. `docs/editorial/seo-article-publication-standard.md` §12): Tier 1 — Яков и Партнёры, НИУ ВШЭ ИСИЭЗ, TAdviser, CNews Analytics. Tier 2 — Sber/SberAI blog, Yandex Research. Tier 3 — Forbes Russia, Ведомости.Технологии. С пометкой «фактчекать»: Habr, vc.ru. Если ни один источник не покрывает факт — переформулировать качественно или оформить нейтральный обезличенный сценарий без fake-precision, не выдумывать.

Запрещено:
— придумывать цифры, тарифы, имена компаний, кейсы клиентов, цитаты;
— ссылаться на ещё не опубликованные гайды как на готовые страницы (`/guides/<future-slug>`);
— использовать FAQPage schema без visible FAQ на странице;
— инфобиз-клише («секрет успешного внедрения», «3 шага к ИИ-трансформации», «прорыв», «революция»);
— дублирующее markdown TOC при наличии sticky aside (render даст «В статье» автоматически);
— general talk «ИИ меняет бизнес» без механики, цифр и кейса.

Объём:
— главные хабы 10 000–18 000 знаков;
— статьи про инструменты 6 000–12 000;
— глоссарные/узкие 2 000–5 000.
Не раздувать ради объёма. Лучше короче и плотнее.
